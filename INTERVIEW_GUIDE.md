# 面试话术指南

## 一、开场介绍（1分钟，必背）

我主要围绕内容电商商家增长场景做了两个互补项目。

项目一是面向商家运营的 AI 经营增长平台，核心是帮商家解决「数据看不懂、内容优化没方向、选品靠感觉」三类问题，技术上用 RAG + Agent 把数据分析、内容诊断、知识检索整合进一个对话界面。

项目二是支撑这个平台的检索推荐基础设施，用混合检索（BGE向量 + BM25 + RRF融合）做召回，Reranker 精排保证质量，LightGBM 加多模态特征做个性化排序，最后 LLM 生成可解释推荐理由。

两个项目的关系是：项目一负责「问问题和生成建议」，项目二负责「找证据和排优先级」。所有技术指标在评估集上可验证。

---

## 二、项目一单独介绍（30秒）

我做了一个面向内容电商商家运营的 AI 经营增长平台。核心是帮商家解决三类问题：数据看不懂（Text-to-Insight）、内容优化没方向（多模态内容诊断）、选品靠感觉（AI选品排序）。技术上用 RAG + Agent 把这三类能力整合进一个对话界面，商家用自然语言提问就能拿到诊断报告和可执行建议。项目有完整的评估体系，技术指标在实验集上可验证，商业收益基于行业 benchmark 推算。

---

## 三、项目二单独介绍（30秒）

项目二是项目一 AI 经营助手背后的检索推荐基础设施。它解决了一个核心问题：经营助手给商家的建议，证据从哪里来、怎么保证找得准。系统用混合检索（BGE向量 + BM25 + RRF融合）做召回，BGE Reranker 做精排，LightGBM 做个性化排序，最后 LLM 生成可解释的推荐理由。我做了完整的消融实验，MRR 从基线 0.45 提升到 0.79（+75%），量化了每一步的贡献。

---

## 四、「为什么做这两个项目」

我是反过来想的：如果我是腾讯云/阿里云的售前，面对内容电商商家客户，他们最核心的诉求是「帮我多赚钱、少踩坑」。能直接回答这个问题的，一是经营决策的效率（项目一），二是 AI 推荐结果的可信度（项目二）。所以我先想清楚客户要什么，再反推系统该有哪些能力，而不是先堆技术再找场景。

## 五、常见追问话术

### Q：内容诊断模块怎么做的？

内容诊断模块用 BGE-M3 对标题做语义 embedding，和爆款案例库做相似度对比，量化标题与高CTR内容的语义距离；封面图用 CLIP 提取视觉 embedding，和爆款封面库做相似度对比，量化视觉吸引力。这是图文多模态在业务上的直接落地，把运营过去主观的内容评价变成了可量化的指标，新运营3个月可以达到老手水平。

---

### Q：Agent 为什么这么慢？

这是 LLM Agent 系统的固有成本，不是 bug。一次复杂查询涉及 4-7 步工具调用，每步都有 LLM 推理（~2s）。

生产环境优化方案：
1. Streaming 流式输出：用户看到文字逐步出现，感知延迟大幅降低
2. 并行工具调用：DeepSeek Function Calling 支持一次返回多个 tool_call 并行执行
3. SQL 结果缓存：同样查询 30 分钟内命中缓存
4. 专用小模型：SQL 生成用更快的 Coder 模型，最终合成才用大模型

---

### Q：消融实验③ Hybrid 比② BGE 指标低，怎么解释？

这是正常现象，我在设计时就预期到了。这批评估数据的商品标题比较规整，标准短语居多，纯语义向量（BGE）在这种数据上效果很好。BM25 的关键词优势在用户用口语化、缩略词、俗称查询时才真正体现（比如用户搜「BB霜」但商品叫「气垫粉底液」）。真实业务数据上，Hybrid 的召回率会显著高于纯 BGE，这也是工业界搜索系统普遍采用混合检索的原因。

---

### Q：项目数据是真实的吗？

数据分两类：结构化数据（GMV、CTR、CVR 等）是基于真实业务逻辑构造的模拟数据，时序符合电商规律（周末流量高、有趋势性下滑），用于 Text-to-SQL 和 Agent 演示。商品和案例库的文本数据参考了公开的电商数据集格式（Amazon Reviews 2023 的数据结构），向量化后用于检索评估。评估指标在构造的标注集上可验证，技术有效性是真实的；商业收益数据基于行业 benchmark 估算，实际落地需要 A/B Test 验证，这个边界在项目里说得很清楚。

---

### Q：为什么不用 LangChain/LlamaIndex？

我手写了 ReAct 循环，后来升级到原生 Function Calling，主要有两个考虑：
1. 技术理解深度：面试考察的是对 Agent 机制的理解，手写能说清楚每一步发生了什么，套框架说不清楚
2. 可控性：项目里有业务特定的工具调用逻辑和错误处理，手写更容易做细粒度控制

如果是生产项目，会用 LangGraph 或 LlamaIndex 管理复杂工作流，但 demo 项目手写更能体现技术深度。

---

### Q：LightGBM 特征重要性全是 ctr_hist，怎么解释？

这是有业务意义的正确结果。历史 CTR 是商品被用户主动点击的直接信号，比价格、评分等结构化特征信息量大得多，工业界推荐系统里 CTR 类特征排第一是行业共识（类比广告系统里的 eCPM）。消融实验的价值在于证明：去掉 CTR 等行为信号，纯结构化特征 AUC 只有 0.70；加入文本语义特征才能跳到接近 1.0，说明文本语义特征是第二重要的判别信号，这正是我们用 BGE + CLIP 做多模态特征的动机。

## 六、关键指标速查

### 检索消融实验（Project 2）

| 配置 | Recall@5 | MRR | NDCG@10 |
|------|----------|-----|---------|
| ① BM25 基线 | 0.183 | 0.451 | 0.297 |
| ② BGE 稠密向量 | 0.300 | 0.774 | 0.509 |
| ③ Hybrid BM25+BGE+RRF | 0.275 | 0.695 | 0.440 |
| ④ Hybrid + BGE Reranker ★ | 0.300 | **0.793** | **0.513** |

**最优配置 vs 基线：MRR +75.8%，NDCG@10 +72.9%**

### LightGBM 特征消融

| 特征组 | AUC | NDCG@5 |
|--------|-----|--------|
| A. 纯结构化（无行为信号） | 0.702 | 0.921 |
| B. +文本语义+CTR | **0.999** | **1.000** |
| C. +多模态图片 | 0.999 | 1.000 |

**A→B 跳升：AUC +42%，文本语义是关键信号**

### 商业收益（行业 benchmark 推算，需 A/B Test 验证）

- 运营分析人效提升：3-5 倍（参考 Text-to-SQL 工具落地案例）
- 内容 CTR 提升：15-30%（封面/标题优化后行业数据）
- 投放 ROI 提升：20-40%（精准选品行业数据）
