## 项目一介绍（4分钟）— AI 经营增长平台

### 为什么是 Agent 架构？

商家的问题是复合的。比如「为什么本周 GMV 下滑」，这一个问题背后需要：
- 查数据（SQL查询漏斗各环节）
- 对比内容质量（封面/标题诊断）
- 找参考案例（爆款检索）
- 给出建议（知识库 + LLM 合成）

如果每个功能是独立页面，商家要来回切换5个工具。所以我用 Agent 把这些工具整合进一个对话框——商家用自然语言提问，Agent 自动拆解任务、编排工具、整合结果。

### 核心模块逐一讲解

**模块1：Text-to-SQL 经营数据分析**

技术链路：自然语言 → DeepSeek 生成 SQL → SQLite 执行 → Plotly 可视化 → LLM 生成洞察

关键设计决策：用 Few-Shot Prompting 而不是 Fine-tune，因为 SQL 的 schema 是固定的，Few-Shot 就够了，不需要训练成本。

**模块2：多模态内容诊断**

封面用 CLIP，标题用 BGE-M3，脚本用 LLM。三者对应三个维度的「内容质量」——视觉、语义、结构。核心思路是：把「爆款案例库」当作基准，量化当前内容与爆款的距离。

**模块3：RAG 运营知识库**

把平台规则、SOP、选品方法论向量化存储，商家问「直播间能引流到微信吗」这类规则性问题时，不靠 LLM 硬记，而是从知识库检索再生成，避免幻觉。

**模块4：ReAct Agent（升级为 Function Calling）**

最初手写 ReAct 循环（Thought→Action→Observation），理解原理后发现 DeepSeek 原生 Function Calling API 更稳定，改为原生 API 调用。核心是：LLM 不直接给答案，而是决定调用哪个工具、拿到数据后再合成答案。

## 项目二介绍（4分钟）— 检索推荐底座

### 为什么需要单独的检索底座？

项目一的 Agent 每次给出建议，背后需要「证据」——用什么案例证明这个标题不够好？推荐哪些商品是有数据依据的？

如果检索做得差，Agent 的回答就是「幻觉」；检索做得好，Agent 才有可信度。所以项目二是项目一的可信度保障。

### 为什么是混合检索？

纯向量检索（BGE）的问题：遇到精确关键词查询时效果差。比如用户搜「SKU018」，向量检索找不到，因为这是ID不是语义。

纯关键词检索（BM25）的问题：语义理解弱。用户搜「夏天通勤防晒」，商品叫「轻薄隔离防紫外线精华」，BM25 召回不到。

RRF 融合：两路并行，各取所长，加权合并，既能处理精确查询又能处理语义查询。

### 为什么要 Reranker？

召回的 Top-50 里排序不一定准。Reranker 用 cross-encoder 对每个「查询-文档」对单独打分，比向量检索的余弦相似度更精准，代价是计算更慢（所以只精排 Top-50 → Top-5，不做全量）。

### LightGBM 的价值

Reranker 只考虑文本相关性。LightGBM 加入业务特征：历史 CTR、评分、类目匹配、图片视觉相似度等。让排序不只考虑「语义像不像」，还考虑「业务上值不值得推」。

### 消融实验

| 配置 | MRR | 相比基线 |
|------|-----|---------|
| ① BM25 基线 | 0.451 | — |
| ② BGE 稠密向量 | 0.774 | +72% |
| ③ Hybrid+RRF | 0.695 | +54% |
| ④ +BGE Reranker | **0.793** | **+76%** |

# 第二部分：每个技术的原理 + 为什么用它

## 2.1 BGE-M3 —— 稠密向量检索

**是什么：** 北京智源（BAAI）开发的多功能多语言向量模型，支持稠密检索、稀疏检索、多向量检索三种模式。

**原理：** 把文本编码为高维向量（1024维），语义相近的文本在向量空间中距离近。检索时计算查询向量与文档向量的余弦相似度。

**为什么选 BGE-M3 而不是 OpenAI Embedding：**
- BGE-M3 中文效果更好（专门在中文语料上训练）
- 可本地部署，不依赖外部 API，无数据隐私风险
- 开源免费，TO B 场景下减少运营成本
- 在 MTEB 中文榜单上排名靠前

**为什么不用更大的模型（如 text-embedding-3-large）：**
1024 维向量对于商品语义匹配已经足够，更大模型计算成本高，且召回质量提升边际递减。

---

## 2.2 BM25 —— 稀疏关键词检索

**是什么：** 基于词频统计的经典检索算法，是 TF-IDF 的改进版本。

**原理：**
```
BM25(q,d) = Σ IDF(qi) × [f(qi,d)×(k1+1)] / [f(qi,d) + k1×(1 - b + b×|d|/avgdl)]
```
- TF（词频）：词在文档中出现越多，分越高
- IDF（逆文档频率）：词在语料中越稀有，分越高
- k1, b：调节参数，k1 控制词频饱和，b 控制文档长度归一化

**为什么不能只用 BM25：**
- 无法理解同义词（「面霜」和「乳霜」不匹配）
- 无法处理语义相似但词汇不同的查询

**为什么不能没有 BM25：**
- 精确关键词查询（SKU ID、品牌名）向量检索效果差
- BM25 计算极快，可以作为第一阶段粗筛

---

## 2.3 RRF —— 倒数排名融合

**是什么：** Reciprocal Rank Fusion，一种无参数的结果融合方法。

**原理：**
```
RRF_score(d) = Σ 1 / (k + rank_i(d))
```
k=60 是平滑常数，防止排名第1的文档权重过大。

**为什么用 RRF 而不是线性加权：**
- 线性加权需要两路分数在同一尺度，向量相似度（0-1）和 BM25 分数（0-∞）尺度不同
- RRF 只看排名，不看分数绝对值，无需归一化，也不需要调参
- 实践证明 RRF 效果通常优于线性融合

---

## 2.4 BGE Reranker —— Cross-encoder 精排

**是什么：** 与 Bi-encoder（双塔）不同，Cross-encoder 把查询和文档一起输入，在注意力层充分交互后输出相关性分数。

**Bi-encoder vs Cross-encoder 的本质区别：**

| | Bi-encoder（BGE-M3） | Cross-encoder（Reranker） |
|---|---|---|
| 原理 | 查询和文档分别编码，余弦相似度 | 拼接后一起编码，输出相关性分 |
| 计算效率 | 高（可预计算文档向量） | 低（每次都要重新计算） |
| 语义理解 | 词级别交互弱 | 词级别充分交互 |
| 用途 | 大规模召回 | 小规模精排 |

**为什么两阶段而不是直接用 Cross-encoder：**
全量文档用 Cross-encoder 计算量是 O(n)，n=500商品时还可以，但工业界百万量级商品根本跑不完。两阶段：Bi-encoder 快速召回 Top-50，Cross-encoder 只精排 50 个，计算可控。

---

## 2.5 CLIP —— 多模态图文向量化

**是什么：** OpenAI 开发的图文对比学习模型，用大量「图片-文字」对训练，使图片和对应描述的向量接近。

**原理：** 对比学习（Contrastive Learning）——正样本（图片和对应描述）向量靠近，负样本（图片和不相关描述）向量远离。

**在项目中的两个应用：**
1. **封面视觉评分**：把封面图编码为向量，和爆款封面库做相似度对比
2. **以图搜货**：用户上传图片，找视觉相似的商品

**为什么不用 ResNet/ViT 直接提特征：**
CLIP 的向量空间是「图文对齐」的，图片和文字描述在同一向量空间。这意味着可以做跨模态检索（文字查图片、图片查文字），这是纯视觉模型做不到的。

---

## 2.6 RAG —— 检索增强生成

**是什么：** Retrieval-Augmented Generation，先从知识库检索相关片段，再把检索结果作为上下文让 LLM 生成答案。

**为什么要 RAG 而不是直接问 LLM：**

| 问题 | 纯 LLM | RAG |
|---|---|---|
| 幻觉 | 高（LLM 编造知识） | 低（基于检索到的事实） |
| 时效性 | 停在训练截止日期 | 实时更新知识库 |
| 可溯源 | 不知道答案从哪来 | 答案带引用来源 |
| 专业知识 | 通用知识，领域知识弱 | 注入领域专属文档 |

**RAG 的核心权衡：**
检索质量直接决定生成质量。「Garbage in, garbage out」——如果召回的片段不相关，LLM 再强也给不出好答案。这就是为什么我在 RAG 里也用了 BGE Reranker 精排，而不是直接用向量检索的 Top-K。

---

## 2.7 LightGBM Learning-to-Rank

**是什么：** 基于梯度提升决策树的排序模型，objective 是 lambdarank（直接优化 NDCG）。

**为什么是 LightGBM 而不是神经网络排序：**
- 可解释性强：SHAP 值直接看特征贡献
- 训练快：梯度提升树比 DNN 快几个数量级
- 小数据也能用：DNN 需要大量样本，LightGBM 几千条就够
- 工业界推荐系统的默认选择（美团/滴滴的 L2R 都用 GBDT）

**NDCG 和 AUC 的区别：**
- AUC：衡量模型把正样本排在负样本前面的能力（二分类视角）
- NDCG：衡量排序质量，越靠前的位置权重越大（考虑位置偏差）
- 推荐场景更关注 NDCG：用户只看前几个结果

---

## 2.8 SHAP —— 模型可解释性

**是什么：** SHapley Additive exPlanations，基于博弈论的特征归因方法。

**原理（Shapley 值）：** 把每个特征看作「玩家」，模型预测看作「收益」，Shapley 值是该特征在所有可能的特征子集中的平均边际贡献。

**为什么需要 SHAP：**
LightGBM 的预测是黑箱，商家不知道为什么推荐了这个商品。SHAP 能说「推荐你这款防晒霜，主要因为它历史 CTR 高（+5.1），其次评论数多（+0.3），价格偏高略有拖累（-0.9）」。让 AI 建议可信、可解释、可采纳。

---

## 2.9 Function Calling —— Agent 工具调用

**是什么：** LLM 原生支持的结构化输出能力，让模型决定是否调用外部函数、调用哪个、传什么参数。

**和手写 ReAct 的区别：**

| | 手写 ReAct | Function Calling |
|---|---|---|
| 原理 | 文本解析（regex 匹配 Action:{}） | 模型原生 JSON 输出 |
| 稳定性 | 模型可能不按格式输出 | 格式由 API 保证 |
| 多工具并行 | 需要多轮 | 一次返回多个 tool_calls |
| 调试难度 | 高（文本解析出错难定位） | 低（结构化 JSON） |

**为什么先手写 ReAct 再改 Function Calling：**
手写让我理解了 Agent 的本质——LLM 在「思考要做什么」和「执行工具」之间的循环。理解原理后再用 API，知道自己在用什么。面试时能说清楚每一步，这比直接套框架更有价值。

# 第四部分：面试高频问题 Q&A

## 技术深度类

**Q：BGE-M3 和 BGE-Large 怎么选择？**

A：BGE-M3 是多功能模型，支持稠密/稀疏/多向量三种检索模式，且在长文本上表现更好（支持 8192 token）；BGE-Large 是纯稠密向量模型，在短文本上速度更快。我们的商品标题一般 20-50 字，两者效果差别不大，选 BGE-M3 是为了后续可以扩展到稀疏检索模式，减少重构成本。

---

**Q：你的 RAG 系统怎么处理知识库更新？**

A：目前是全量重建：知识库文档更新后，重新向量化建索引。生产环境应该做增量更新：只对新增/修改的文档重新向量化，删除旧向量，插入新向量。FAISS 支持 `remove_ids` 和 `add` 操作，可以实现增量更新。另外需要维护文档 ID 到向量 ID 的映射表。

---

**Q：你的混合检索为什么 Hybrid③ 比纯 BGE② 的 MRR 低？**

A：这是数据特性决定的。我的评估集商品标题比较规整（标准商品名），纯语义向量已经足够准确。BM25 在这种场景下加入噪音，RRF 融合后反而稀释了 BGE 的精准性。真实业务场景里用户查询更口语化、更模糊，这时 Hybrid 的优势会体现出来。这也是工业界搜索系统标配 Hybrid 的原因——线上分布和评估集不同。

---

**Q：你的 LightGBM 消融实验 B 和 C 都是 1.0，这个结果可信吗？**

A：这里要坦诚。训练数据是用 CTR 构造的模拟标签，B 组加入了 CTR 相关特征（rerank_score 等），相当于直接用了标签信号，所以 AUC 接近完美。这在真实场景不会出现——真实场景里特征和标签有噪声。消融实验的核心价值在于 A 到 B 的对比：纯结构化特征（无行为信号）AUC 只有 0.70，加入文本语义特征后跳到 1.0，说明文本语义特征是关键判别维度，这个结论本身是有意义的。

---

**Q：你的 Agent 为什么不用 LangChain？**

A：两个原因。第一，理解原理：面试考察的是对 Agent 机制的理解，手写让我能解释每一步发生了什么——LLM 如何决定调用哪个工具、工具返回结果后如何合成答案。第二，控制性：LangChain 的抽象层在工具 schema 定义和错误处理上灵活性差。项目里后来确实升级为 DeepSeek 原生 Function Calling，比手写 ReAct 更稳定。生产环境我会选 LangGraph 管理复杂工作流状态。

---

**Q：FAISS 和 Elasticsearch 怎么选？**

A：取决于场景。FAISS 是纯向量检索库，不带服务化，适合嵌入 Python 进程、中小规模（百万量级以内）、延迟敏感场景。Elasticsearch 是完整搜索引擎，带 HTTP 接口、分布式、混合检索（dense+sparse）、过滤、实时更新，适合生产环境、需要运维界面、团队已有 ES 基础。我用 FAISS 是因为 Demo 场景，生产环境推荐 ES 8.x 或 OpenSearch，已经原生支持 kNN 向量检索。

---

**Q：Reranker 的延迟怎么控制？**

A：BGE Reranker 对每个 query-doc 对做 full attention，延迟是 O(n×seq_len²)。在我的项目里 Top-50 精排，单次约 500ms-1s（CPU）。生产环境优化方向：
1. 用 GPU，延迟降 10 倍
2. 模型量化（INT8），精度损失约 1%，速度提升 2-4 倍
3. 减少精排候选（Top-20 而不是 Top-50）
4. 用更小的 Reranker 模型（如 bge-reranker-base）

---

**Q：你的评估集是怎么构建的？**

A：分两步。第一步生成候选：从商品标题提取自然语言查询（改写标题中的关键词）。第二步标注相关性：同类目商品视为相关（relevance=1），查询商品本身视为强相关（relevance=2）。这是弱监督方式，没有人工标注每个查询对的相关性。局限性是标注可能不精准——类目相同不代表语义相关。生产环境应该用人工标注的黄金评估集，或者收集用户点击行为作为隐式反馈。

## 产品思维类

**Q：这个产品上线后如何衡量成功？**

A：核心指标分三层：
- **北极星指标**：商家续约率（衡量产品是否真正创造价值）
- **行为指标**：月活商家数、每天平均对话轮次、各功能使用率
- **业务指标**：使用我们产品的商家 GMV 同比增长率（对照实验）

A/B Test 设计：随机抽取商家，50% 使用 AI 助手，50% 对照组，观察 90 天的 GMV 变化、内容 CTR 变化、运营人效变化。

---

**Q：如果商家的数据接不进来，产品怎么用？**

A：这是 MVP 最大的障碍。数据接入有几个路径：
1. **平台官方 API**：抖音、淘宝都有商家数据 API，授权后可拉取
2. **人工导出**：商家从后台下载 CSV，上传到平台
3. **爬虫辅助**：抓取公开的竞品数据作为基准
4. **冷启动策略**：无数据时，先用行业 benchmark 数据，用户用着用着积累了自己的数据再切换

---

**Q：为什么不直接做 ToC，面向商家个人？**

A：有三个原因选择 ToB（卖给云厂商）：
1. **数据获取**：云厂商已有商家数据授权，直接 ToC 需要自己搭数据链路
2. **信任背书**：中小商家更信任腾讯云/阿里云的品牌，而不是一家新创公司
3. **分发效率**：云厂商的销售网络覆盖了目标商家，自己做销售成本极高

---

**Q：这个产品的护城河是什么？**

A：短期护城河是**数据飞轮**：接入的商家越多，爆款案例库越丰富，内容诊断越准确，吸引更多商家，形成正循环。长期护城河是**行业数据积累**：细分行业的运营知识（美妆的爆款逻辑和食品的不一样）形成差异化知识库，纯通用 LLM 复制不了。

---

## 系统设计类

**Q：如果 QPS 要支持 10000，系统怎么改造？**

A：当前是单进程 Demo，生产化改造方向：

1. **离线计算**：商品向量化离线预计算，存入向量数据库（Milvus/Pinecone）
2. **模型服务化**：BGE-M3 和 Reranker 用 TorchServe/Triton 独立部署，水平扩展
3. **缓存层**：高频查询 Redis 缓存，命中率约 30-40%
4. **异步 Agent**：Agent 任务队列化，WebSocket 推送结果，避免 HTTP 超时
5. **限流**：针对 LLM 调用做令牌桶限流（LLM API 是瓶颈）

---

**Q：你的数据库选型是什么？为什么用 SQLite？**

A：SQLite 是 Demo 选择，轻量、无需部署、适合单机场景。生产环境根据数据规模选择：
- **< 1亿行**：MySQL/PostgreSQL + 列式存储扩展
- **> 1亿行**：ClickHouse（OLAP 列式存储，聚合查询快 100 倍）
- **实时数据**：Kafka → Flink → ClickHouse，支持实时漏斗分析

Text-to-SQL 的 schema 设计也需要优化：生产环境的表结构更复杂（50+个表），Few-Shot 样本不够，需要 Schema Linking（先从几百张表里找相关表，再生成 SQL）。

# 第五部分：常见陷阱 & 主动坦诚

以下问题面试官可能直接问，也可能在追问中触及，建议主动坦诚：

1. **数据是模拟的**：是的，基于真实业务逻辑构造，核心指标趋势符合真实场景，但没有真实用户验证。

2. **LightGBM AUC=1.0 是数据泄露**：训练标签和测试特征来自同一分布（CTR），消融价值在 A vs B 对比，而非 B 的绝对值。

3. **没有并发压测**：没有做过 QPS 测试，只是 Demo 级别，生产级别需要改造。

4. **以图搜货 demo 模式不可用**：demo 模式无图片向量，full 模式（AutoDL GPU）可用。

5. **Hybrid MRR 低于纯 BGE**：数据特性问题（规整标题），真实场景不一样。

**坦诚的好处：** 主动说出局限性，证明你对项目的理解是深入的，而不是背话术。面试官最怕的是候选人连自己项目的 bug 都不知道。

---

*最后一条建议：技术细节背熟了，但面试时最打动人的往往是你能说清楚「为什么这么做」，而不是「做了什么」。每个技术选型背后都有 trade-off，能把 trade-off 说清楚，就是技术深度。*
